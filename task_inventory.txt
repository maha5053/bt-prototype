Это задача для бекендера и фронтендера на реализацию страницы «Инвентаризация» и её функционала на основе прототипа, разработанного в текущем проекте (папке). dev tools не должны попадать в постановку задачи. Путь к кликабельному прототипу берется из URL GitHub, открывающего нужную страницу.

Саммари задачи

Реализовать в системе страницу «Инвентаризация» — проведение сплошной инвентаризации склада: сравнение системных остатков с фактическим количеством, фиксация расхождений, формирование документа инвентаризации. Документ проходит жизненный цикл: черновик → завершён. В рамках сессии пользователь вводит фактическое количество по каждой позиции (номенклатура + лот + место хранения), система автоматически рассчитывает расхождение и определяет статус: совпадение / излишек / недостача.

1. Модель данных «Сессия инвентаризации»

Сессия инвентаризации — это документ, содержащий шапку (дата создания, создатель, статус) и список позиций для сверки.

1.1. Структура сессии

Поле | Тип | Описание
--- | --- | ---
id | string | Уникальный идентификатор сессии
createdAt | datetime | Дата и время создания
createdBy | string | ФИО пользователя, создавшего сессию
completedAt | datetime? | Дата и время завершения (null для черновика)
status | enum | "draft" / "completed"
lines | InventoryLine[] | Список позиций для сверки

1.2. Структура позиции (InventoryLine)

Поле | Тип | Описание
--- | --- | ---
nomenclatureId | string | Идентификатор номенклатуры
nomenclatureName | string | Наименование номенклатуры
group | string | Группа номенклатуры (Проточная цитометрия, ИФА, Расходники для ЦФА и т.д.)
manufacturer | string | Производитель
lot | string | Номер партии
place | string | Место хранения
systemQuantity | number | Системное количество (остаток на момент создания сессии)
actualQuantity | number? | Фактическое количество (null — не проверено)
discrepancy | number? | Расхождение (actualQuantity − systemQuantity, null если не проверено)
status | enum | "совпадение" / "излишек" / "недостача"
comment | string | Комментарий к позиции

1.3. Правила расчёта расхождений

- Если `actualQuantity === null` → `discrepancy = null`, статус не определяется (позиция не проверена).
- Если `actualQuantity === systemQuantity` → `discrepancy = 0`, статус «совпадение».
- Если `actualQuantity > systemQuantity` → `discrepancy > 0`, статус «излишек».
- Если `actualQuantity < systemQuantity` → `discrepancy < 0`, статус «недостача`.

1.4. Правила формирования списка позиций

- При создании новой сессии в неё включаются **все позиции** из текущих остатков склада (номенклатура + лот + место хранения).
- `actualQuantity` для всех новых позиций устанавливается в `null` (не проверено).
- Если существует сессия в статусе «draft», при создании новой она автоматически завершается (статус → «completed»).

2. Страница «Инвентаризация» — Журнал сессий

Доступна для всех ролей.

2.1. Структура страницы

Страница состоит из:

- заголовка «Инвентаризация»;
- кнопки «Начать инвентаризацию» (создаёт новую сессию и переходит на страницу детальной сессии);
- таблицы документов инвентаризации.

2.2. Таблица документов

Каждая строка — один документ инвентаризации. Столбцы:

Столбец | Описание
--- | ---
Номер | Порядковый номер документа (ссылка на детальную страницу сессии)
Дата создания | Дата и время создания (формат ДД.ММ.ГГГГ ЧЧ:ММ)
Создатель | ФИО пользователя, создавшего документ
Статус | Бейдж: «В процессе» (для черновика) / «Завершена» (для завершённой)
Позиций | Общее количество позиций в сессии
Проверено | Количество позиций с введённым фактическим количеством
Завершена | Дата завершения (если применимо)

2.3. Пустое состояние

Если документов нет — отображается заглушка с предложением начать первую инвентаризацию.

3. Создание сессии инвентаризации

При нажатии «Начать инвентаризацию»:

1. Создаётся новая сессия со статусом «draft».
2. В неё загружаются все текущие остатки склада (номенклатура + лот + место хранения).
3. Все позиции получают `actualQuantity = null` (не проверено).
4. Если существовала сессия в статусе «draft» — она автоматически завершается.
5. Происходит переход на детальную страницу сессии.

4. Детальная страница сессии инвентаризации

Отображает полную информацию о документе и позволяет вводить фактические количества.

4.1. Содержимое

- Заголовок: «Инвентаризация».
- Дата и время создания.
- Создатель (ФИО).
- Статус (бейдж: «В процессе» / «Завершена»).
- Карточки-сводка (summary):
  - Всего позиций — общее количество позиций в сессии.
  - Не проверено — количество позиций с `actualQuantity = null`.
  - Совпадения — количество позиций со статусом «совпадение».
  - Излишки — количество позиций со статусом «излишек».
  - Недостачи — количество позиций со статусом «недостача».

4.2. Таблица позиций

Каждая строка содержит:

Столбец | Описание | Редактирование
--- | --- | ---
Номенклатура | Название + группа · производитель | Нет
Партия | Номер партии | Нет
Место хранения | Место | Нет
Системное кол-во | Остаток на момент создания | Нет
Фактическое кол-во | Введённое количество | Да (для черновика)
Расхождение | Разница (факт − система) | Авто, нет
Статус | Бейдж: не проверено / совпадение / излишек / недостача | Авто, нет
Комментарий | Текстовое поле | Да (для черновика)

4.3. Поведение ячеек «Фактическое кол-во»

- **Не проверено** (`actualQuantity = null`): отображается «—» в рамке с пунктирной оранжевой обводкой. При клике — поле ввода.
- **Проверено**: отображается введённое число. При клике — поле ввода для редактирования.
- Поле ввода: числовое поле, минимум 0. Сохранение по Enter или потере фокуса (blur). Отмена по Escape.
- При вводе автоматически пересчитываются `discrepancy` и `status`.
- Для завершённых сессий ячейки нередактируемы — отображаются как plain text.

4.4. Поведение ячейки «Расхождение»

- Если `actualQuantity = null` — отображается «—» серым цветом.
- Если `discrepancy > 0` — синим цветом с знаком `+`.
- Если `discrepancy < 0` — красным цветом с знаком `−`.
- Если `discrepancy = 0` — зелёным цветом.

4.5. Поведение ячейки «Статус»

- **Не проверено** — серый бейдж (отображается только при `actualQuantity = null`).
- **Совпадение** — зелёный бейдж.
- **Излишек** — синий бейдж.
- **Недостача** — красный бейдж.

4.6. Поиск и фильтрация

Строка поиска и выпадающие списки фильтров над таблицей:

Фильтр | Тип | Значения
--- | --- | ---
Поиск | Текстовое поле | Поиск по названию номенклатуры, партии, производителю (частичное совпадение)
Место хранения | Выпадающий список | Все места / конкретные места из системы
Группа номенклатуры | Выпадающий список | Все группы / конкретные группы
Статус расхождения | Выпадающий список | Все статусы / Не проверено / Совпадение / Излишек / Недостача

- Кнопка «Сбросить» — сбрасывает все фильтры и поиск.
- Фильтрация применяется к отображаемым строкам, пагинация пересчитывается.

4.7. Пагинация

- Размер страницы: 15 строк.
- Навигация: «Назад [номер страницы] Вперёд».
- Отображение диапазона: «Показано X–Y из Z».

5. Действия по сессии

Доступны только для сессий в статусе «draft».

5.1. Сохранить черновик

- Кнопка «Сохранить черновик» — явное сохранение текущего состояния.
- После нажатия отображается toast-уведомление (bottom-right) с информацией:
  - Время сохранения (формат ЧЧ:ММ).
  - Количество проверенных и непроверенных позиций.
  - Пример: `14:23 · Проверено 45, не проверено 120 из 165`.
- Уведомление автоматически исчезает через 4 секунды.

5.2. Завершить сессию

- Кнопка «Завершить» — переводит сессию в статус «completed».
- При нажатии показывается диалог подтверждения:
  - «Вы уверены, что хотите завершить инвентаризацию? Все непроверенные позиции останутся с фактическим количеством "не указано". После завершения редактирование будет недоступно.»
- После подтверждения:
  - Статус → «completed».
  - Фиксируется `completedAt`.
  - Кнопки действий скрываются.
  - Все ячейки становятся нередактируемыми.
  - Отображается бейдж «Завершена».

5.3. Удалить черновик

- Кнопка «Удалить» (иконка корзины) — доступна только для черновиков.
- При нажатии показывается модальное окно подтверждения с красным предупреждением:
  - «Вы уверены, что хотите удалить черновик инвентаризации? Это действие нельзя отменить.»
- После подтверждения сессия удаляется, происходит переход на страницу журнала.
- Для завершённых сессий кнопка удаления не отображается.

6. Хранение и нумерация

- Сессии инвентаризации хранятся в системе с уникальным `id`.
- Нумерация документов — сквозная, порядковая.
- Каждая сессия фиксирует `createdAt` — дату и время создания.
- Создатель — пользователь, создавший сессию.
- Данные сессии сохраняются при каждом изменении (явное сохранение черновика или авто-сохранение при вводе).

7. Фронтенд — требования к интерфейсу

7.1. Список сессий

- Таблица сессий с колонками: номер, дата, создатель, статус, количество позиций, прогресс проверки.
- Кнопка «Начать инвентаризацию» — создаёт новую сессию, автозавершает существующий черновик.
- Пустое состояние при отсутствии документов.
- Клик по номеру сессии — переход на детальную страницу.

7.2. Детальная сессия

- Шапка с датой, создателем, статусом.
- Карточки-сводка с количеством позиций по статусам.
- Панель поиска и фильтрации (по месту, группе, статусу расхождения).
- Таблица позиций с inline-редактированием фактического количества и комментария.
- Пагинация (15 строк на странице).
- Кнопки действий: «Сохранить черновик», «Завершить», «Удалить» (только для draft).
- Toast-уведомление при сохранении черновика.
- Модальные окна подтверждения для завершения и удаления.

7.3. Стили и UX

- Ячейки «не проверено» — пунктирная рамка оранжевого цвета с «—».
- Статусные бейджи цветные: серый (не проверено), зелёный (совпадение), синий (излишек), красный (недостача).
- Расхождения подсвечиваются: синий `+N` (излишек), красный `−N` (недостача), зелёный `0` (совпадение).
- Редактируемые ячейки при клике превращаются в input с autofocus.
- Сохранение по Enter или blur, отмена по Escape.
- Завершённые сессии полностью read-only — без кнопок и редактирования.
- Адаптивная сводка: на мобильных — 2 колонки, на десктопе — 5.

8. Бекенд — требования к API

8.1. Эндпоинты

Метод | Путь | Описание
--- | --- | ---
GET | `/api/inventory` | Получить список всех сессий (краткая информация)
POST | `/api/inventory` | Создать новую сессию (автозавершение существующего draft)
GET | `/api/inventory/{id}` | Получить детальную информацию о сессии
PUT | `/api/inventory/{id}/lines/{lineIndex}/actual` | Обновить фактическое количество
PUT | `/api/inventory/{id}/lines/{lineIndex}/comment` | Обновить комментарий к позиции
POST | `/api/inventory/{id}/complete` | Завершить сессию
POST | `/api/inventory/{id}/save-draft` | Явно сохранить черновик (возвращает статистику)
DELETE | `/api/inventory/{id}` | Удалить черновик (только статус draft)

8.2. Бизнес-логика

- При создании сессии фиксируются текущие остатки склада — snapshot (копии данных на момент создания).
- При создании новой сессии существующий draft автоматически завершается.
- Расчёт `discrepancy` и `status` производится на сервере при обновлении `actualQuantity`.
- Завершённую сессию нельзя изменить или удалить.
- Удаление возможно только для сессий в статусе «draft».

8.3. Валидация

- `actualQuantity` ≥ 0.
- `comment` — макс. 500 символов.
- Завершение сессии: статус должен быть «draft».
- Удаление: статус должен быть «draft».

9. Критерии приёмки

- Пользователь может создать новую сессию инвентаризации, которая включает все текущие остатки склада.
- При создании новой сессии существующий черновик автоматически завершается.
- Все новые позиции имеют `actualQuantity = null` (не проверено).
- Пользователь может вводить и редактировать фактическое количество inline (Enter/blur для сохранения, Escape для отмены).
- Расхождение и статус пересчитываются автоматически при вводе.
- Ячейки «не проверено» отображаются с пунктирной оранжевой рамкой и «—».
- Статусные бейджи цветные: серый, зелёный, синий, красный.
- Пользователь может фильтровать позиции по месту, группе и статусу расхождения.
- Пользователь может искать позиции по названию, партии, производителю.
- Пагинация работает (15 строк на странице).
- Карточки-сводка отображают корректные количества по всем статусам.
- Пользователь может сохранить черновик с toast-уведомлением (время, проверено/не проверено/всего).
- Пользователь может завершить сессию с подтверждением — после завершения редактирование недоступно.
- Пользователь может удалить черновик с подтверждением — завершённые сессии удалить нельзя.
- Журнал сессий отображает все сессии со статусами, датами и прогрессом проверки.
- Завершённые сессии полностью read-only.
